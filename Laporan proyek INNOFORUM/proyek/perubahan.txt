Analisa & Saran Optimasi Tipe Data
1. Tabel Users (kampus_forum_users)
username varchar(255) → Cukup 50 (jarang username lebih dari 32 karakter)
role varchar(255) di gambar, di deskripsi enum → Rekomendasi ENUM ('admin','mahasiswa','dosen')
name varchar(255) → Cukup 100 (nama orang jarang lebih)
email varchar(255) → Sudah standar
photo, avatar varchar(255) → Sudah ok (path/file)
prodi varchar(255) → Cukup 100 (nama prodi jarang panjang)
otp_code varchar(255) → Biasanya kode OTP < 20 karakter, tapi biarkan saja untuk fleksibilitas
password varchar(255) → Sudah standar hash bcrypt
Kesimpulan: Banyak yang bisa dipangkas, tapi 255 juga tidak masalah untuk Laravel. Kalau mau hemat space, bisa dipangkas.

2. Tabel User Profiles
prodi & fakultas & angkatan varchar(255) → Bisa 100 (nama prodi/fakultas pendek, angkatan 10 saja cukup)
bio text → Sudah pas

3. Tabel Admin Actions
action_type varchar(255) → Cukup 50 (isi: create, update, delete, dst)
target_type varchar(255) → Cukup 100

4. Tabel Questions
title varchar(255) → Cukup 100 (judul pertanyaan biasanya tidak panjang)
images text → Sudah ok (kalau perlu bisa array json)
content text → Pas

5. Tabel Question Likes
user_id, question_id → Sudah benar
id (PK) → Sudah benar, bisa juga pakai composite (question_id, user_id) kalau ingin 1 user hanya 1 like per pertanyaan

6. Tabel Hashtags
name varchar(255) → Cukup 50-100 (hashtag jarang panjang)

7. Tabel Hashtag Questions
Sudah benar

8. Tabel Comments
content text → Sudah benar
image varchar(255) → Sudah benar

9. Tabel Answers
Sudah benar

10. Tabel Categories
name varchar(255) → Cukup 100

11. Tabel Announcement
title varchar(255) → Cukup 100
content text → Sudah benar

12. Tabel Notifications
type varchar(255) → Cukup 50 (jenis notif sedikit)
data text → Sudah benar

13. Tabel Reports
reason varchar(255) → Cukup 100
status varchar(255) di gambar, di deskripsi enum → Rekomendasi ENUM ('pending','reviewed','resolved','ignored')











Rekomendasi Panjang Kolom (Best Practice & Real Case)
1. email di users
Tetap pakai varchar(255)
Kenapa? RFC email membolehkan alamat email hingga 254 karakter. Ini sudah best practice Laravel & framework lain.
Jangan diperkecil! Kadang ada user dengan alamat email sangat panjang.
2. password di users
Tetap pakai varchar(255)
Hash bcrypt/argon2 seringkali hasilnya 60–100 karakter, kadang hash lain bisa lebih dari itu.
JANGAN diperkecil, karena suatu saat kamu bisa ganti algoritma hash yang lebih panjang.
3. username
Cukup varchar(50)
Hampir tidak ada username yang realistis lebih dari 30 karakter.
4. name (nama user)
Cukup varchar(100)
Nama Indonesia jarang lebih dari 60 karakter; 100 sudah sangat aman.
5. prodi/fakultas
Cukup varchar(100)
Nama fakultas/prodi di Indonesia biasanya pendek.
6. photo/avatar
Tetap varchar(255) (path/file image)
Karena path bisa panjang, atau jika pakai url CDN/foto.
7. otp_code
Cukup varchar(20)
OTP biasanya 4–8 digit, kadang untuk kode unik/token bisa 20 karakter.
8. angkatan
Cukup varchar(10)
Misal: "2022", "2021/2022", "2020", dsb.
9. role
ENUM: 'admin', 'mahasiswa', 'dosen'
10. status (untuk report dsb)
ENUM: 'pending', 'reviewed', 'resolved', 'ignored'
